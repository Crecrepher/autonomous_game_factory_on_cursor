# Join And Merge Review — AGF v3.0

> 병렬 빌드 완료 후 합류, 충돌 감지, 리뷰, 머지 프로세스

---

## 1. 개요

병렬로 실행된 TaskExecutionUnit들이 완료되면 결과를 수집하고,
충돌을 감지하고, 리뷰하고, 최종 머지하는 단계.

```
Parallel Build Phase
  ├── Task A: done ✅
  ├── Task B: done ✅
  └── Task C: done ✅
       ↓
  [Join] → 결과 수집
       ↓
  [Conflict Detection] → 충돌 감지
       ↓
  [Merge Readiness Check] → 머지 가능 여부 판단
       ↓
  [Reviewer Validation] → 코드 품질 검증
       ↓
  [Human Gate] → 사람 검증
       ↓
  [Merge & Commit] → 최종 반영
```

---

## 2. Join — 결과 수집

### 2.1 수집 대상

| 항목 | 출처 |
|---|---|
| 생성/수정된 파일 목록 | 각 TaskExecutionUnit.result.files_changed |
| 검증 결과 | 각 TaskExecutionUnit.result.validation_report_path |
| 에이전트 슬롯 보고 | AgentLease.status + 실행 로그 |
| 실패/타임아웃 태스크 | AgentLease.status == failed/timeout |

### 2.2 Join 조건

| 조건 | 필수 여부 |
|---|---|
| feature_group 내 모든 태스크가 done 또는 blocked | 필수 |
| 활성 임대가 0개 | 필수 |
| blocked 태스크가 있으면 보고에 포함 | 필수 |

### 2.3 Join 결과 구조

```yaml
join_result:
  feature_group: "economy_system"
  timestamp: "2026-03-18T11:00:00Z"
  completed_tasks:
    - name: CurrencyWallet
      files_changed: [ICurrencyWallet.cs, CurrencyWalletConfig.cs, ...]
      validation: pass
    - name: ShopSystem
      files_changed: [IShopSystem.cs, ...]
      validation: pass
  blocked_tasks: []
  failed_tasks: []
  total_files_changed: 12
  potential_conflicts: []
```

## 3. Conflict Detection — 충돌 감지

### 3.1 충돌 유형

| 유형 | 설명 | 심각도 |
|---|---|---|
| File Overlap | 두 태스크가 동일 파일 수정 | CRITICAL |
| Shared Interface Clash | Shared/ 인터페이스를 두 태스크가 수정 | HIGH |
| Namespace Collision | 동일 네임스페이스에 동일 타입 생성 | HIGH |
| Dependency Mismatch | 참조하는 인터페이스 시그니처 불일치 | MEDIUM |
| Config Key Collision | ScriptableObject 필드 이름 충돌 | LOW |

### 3.2 감지 로직

```
FOR each pair (TaskA, TaskB) in completed_tasks:
    filesA = TaskA.files_changed
    filesB = TaskB.files_changed

    overlap = intersection(filesA, filesB)
    IF overlap is not empty:
        report CRITICAL conflict

    IF TaskA modifies Shared/ AND TaskB modifies Shared/:
        report HIGH conflict

    namespacesA = extract_namespaces(TaskA)
    namespacesB = extract_namespaces(TaskB)
    IF intersection(namespacesA, namespacesB) has duplicate types:
        report HIGH conflict
```

### 3.3 충돌 해결

| 심각도 | 자동 해결 | 수동 개입 |
|---|---|---|
| CRITICAL | 불가 → 머지 차단 | 사람이 직접 수정 |
| HIGH | 불가 → 머지 차단 | 사람이 확인 후 결정 |
| MEDIUM | 가능 (인터페이스 버전 조정) | 사람 확인 권장 |
| LOW | 가능 (자동 리네임) | 보고만 |

## 4. Merge Readiness Check

### 4.1 체크리스트

| # | 항목 | 조건 |
|---|---|---|
| 1 | 모든 완료 태스크의 validation == pass | 필수 |
| 2 | CRITICAL/HIGH 충돌 없음 | 필수 |
| 3 | blocked 태스크가 feature_group 완결에 영향 없음 | 필수 |
| 4 | 모듈 구조 검증 (6파일 완전성) | 필수 |
| 5 | MODULE_REGISTRY 엔트리 존재 | 필수 |
| 6 | TASK_QUEUE 상태 일관성 | 필수 |
| 7 | 테스트 최소 1개 존재 | 필수 |

### 4.2 결과

```yaml
merge_readiness:
  feature_group: "economy_system"
  ready: true | false
  blockers:
    - "ShopSystem: test file missing"
    - "CurrencyWallet × ShopSystem: Shared/ICurrency.cs conflict"
  warnings:
    - "HealthSystem: low priority config key overlap"
```

## 5. Reviewer Validation

### 5.1 리뷰 범위

각 완료된 태스크에 대해:

1. **모듈 구조**: 6파일 완전성, 네이밍 규칙
2. **코딩 규칙**: CODING_RULES.md 준수 (GC 없음, foreach 없음, 매직넘버 없음 등)
3. **의존성 정합성**: MODULE_REGISTRY의 dependencies와 실제 코드 참조 일치
4. **인터페이스 준수**: IModule 시그니처 준수
5. **테스트 존재**: 최소 1개 NUnit 테스트

### 5.2 교차 리뷰 (병렬 작업 특유)

병렬 빌드된 모듈 간:

1. **인터페이스 호환**: 서로 참조하는 인터페이스가 양쪽에서 일관적인지
2. **의존 방향**: 순환 의존 없는지
3. **이벤트 시그니처**: 이벤트 발행/구독 타입 일치

```

Completed: [CurrencyWallet, ShopSystem, RewardSystem]

Merge Ready: YES

═══ 검증 대상 ═══
1. Assets/Game/Modules/CurrencyWallet/ (6 files)
2. Assets/Game/Modules/ShopSystem/ (6 files)
3. Assets/Game/Modules/RewardSystem/ (6 files)

═══ 액션 ═══
Unity Editor → Tools/AI/Validate Generated Modules
```

### 6.2 Human Gate 이후

- `human_state: pending → validated` (각 태스크)
- 모든 태스크 validated → Commit 가능

## 7. Merge & Commit

### 7.1 머지 순서

워크트리/브랜치 기반인 경우:

```
1. agf/build/economy/CurrencyWallet → ai_test (fast-forward 또는 merge)
2. agf/build/economy/ShopSystem → ai_test
3. agf/build/economy/RewardSystem → ai_test
4. 충돌 발생 시 중단 + 사람 개입
```

L0 폴더 격리의 경우:
- 이미 동일 워크트리에서 작업 → 별도 머지 불필요
- YAML 파일만 오케스트레이터가 일괄 업데이트

### 7.2 커밋 메시지

```
feat(economy_system): add CurrencyWallet, ShopSystem, RewardSystem modules

- modules:
  - CurrencyWallet (parallel build, slot 1)
  - ShopSystem (parallel build, slot 2)
  - RewardSystem (parallel build, slot 3)
- parallel_execution: true
- execution_slots: 3
- conflicts: 0
- validation: PASS
- human_validated: true
- generated by: Autonomous Game Factory v3.0
```

## 8. 실패 복구

### 8.1 태스크 실패 시

```
IF task.result.status == failed:
    1. AgentLease 해제
    2. task.status → blocked
    3. task.blocked_reason 기록
    4. 나머지 태스크는 계속 진행 (의존성 없으면)
    5. Join 시 blocked 태스크 별도 보고
```

### 8.2 타임아웃 시

```
IF task.result.status == timeout:
    1. AgentLease 자동 해제
    2. task.status → ready (재시도 가능)
    3. retry_count += 1
    4. IF retry_count >= MAX_RETRY: task.status → blocked
```

### 8.3 충돌 시 (머지 단계)

```
IF conflict.severity >= HIGH:
    1. 머지 중단
    2. 모든 충돌 보고
    3. 사람 개입 대기
    4. 충돌 해결 후 재시도
```
