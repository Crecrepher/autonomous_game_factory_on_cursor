# 프레임워크 설계 원칙

AI Game Dev Framework의 설계 결정과 그 이유를 설명합니다.

---

## 1. 핵심 철학

### "AI는 도구이고, 규칙은 사람이 만든다"

- AI가 자율적으로 규칙을 학습하거나 변경하지 않습니다.
- 모든 규칙(코딩, 구조, 워크플로)은 **사람이 명시적으로 문서화**합니다.
- AI는 그 문서를 읽고 따릅니다.
- 개선도 사람이 문서·검증 규칙을 수정하는 방식으로만 이루어집니다.

### "안전하고 점진적으로"

- 한 번에 많은 코드를 바꾸지 않습니다.
- 한 기능 = 한 모듈 단위로 개발·검증·확인합니다.
- Core 등 위험 영역은 명시적 허용 없이 건드리지 않습니다.

---

## 2. 모듈 아키텍처

### 모듈 = 독립된 기능 단위

각 모듈은 다음 5가지 파일로 구성됩니다:

| 파일 | 역할 | 상속 |
|------|------|------|
| `I<Module>.cs` | 공개 계약 (인터페이스) | - |
| `<Module>Config.cs` | 설정 데이터 | `ScriptableObject` |
| `<Module>Runtime.cs` | 순수 비즈니스 로직 | 없음 (순수 C#) |
| `<Module>Factory.cs` | Config → Runtime 생성 | `static class` |
| `<Module>Bootstrap.cs` | 씬 진입점 | `MonoBehaviour` (얇게) |

### 왜 이 구조인가?

1. **테스트 가능성**: Runtime이 MonoBehaviour를 상속하지 않으므로, 순수 유닛 테스트가 가능합니다.
2. **관심사 분리**: 설정(Config), 로직(Runtime), 생성(Factory), 바인딩(Bootstrap)이 분리됩니다.
3. **의존성 주입**: Factory를 통해 Runtime을 생성하므로, 테스트에서 Mock Config를 주입할 수 있습니다.
4. **AI 친화적**: 패턴이 일정하므로, AI가 새 모듈을 만들 때 템플릿을 복사하기만 하면 됩니다.

---

## 3. 검증 시스템

### 왜 검증이 필요한가?

AI가 생성한 코드가 프로젝트 규칙을 지키는지 자동으로 확인해야 합니다.
사람이 일일이 리뷰하는 것보다, 기계적 규칙은 자동 검증이 효율적입니다.

### 검증기 설계 원칙

1. **인터페이스 기반**: 모든 검증기는 `IModuleValidator`를 구현합니다. 새 검증기를 추가할 때 기존 코드를 수정할 필요가 없습니다 (개방-폐쇄 원칙).
2. **독립적**: 각 검증기는 다른 검증기에 의존하지 않습니다.
3. **보고서 누적**: 하나의 `ValidationReport`에 모든 검증기가 결과를 누적합니다.
4. **Error vs Warning**: Error는 반드시 수정해야 하는 위반, Warning은 참고 사항입니다.

---

## 4. AI 개발 루프

```
문서 읽기 → 계획 → 코드 생성 → 검증 → 요약 → (실패 시) 개선 → 재검증
```

### 왜 이 순서인가?

1. **문서 먼저**: AI가 프로젝트 맥락을 이해한 뒤에 코드를 작성해야 올바른 결과가 나옵니다.
2. **계획 먼저**: 무계획 코드 생성은 구조를 깨뜨립니다.
3. **검증 필수**: 코드 생성 후 반드시 검증을 거쳐야 품질이 보장됩니다.
4. **요약 필수**: 변경사항과 리스크를 명시적으로 기록해야 추적이 가능합니다.

---

## 5. 프로젝트 독립성

### 프레임워크가 강제하는 것

- 모듈 구조 패턴 (Interface, Config, Runtime, Factory, Bootstrap, Tests)
- AI 워크플로 (문서 → 계획 → 구현 → 검증 → 요약)
- 검증 인프라 (ValidationRunner, Validators)

### 프레임워크가 강제하지 않는 것

- 게임 장르, 게임플레이 루프
- 구체적인 코딩 규칙 (네이밍 등은 예시로 제공, 프로젝트가 재정의)
- 모듈 목록 (프로젝트가 MODULE_REGISTRY.yaml로 정의)
- 네임스페이스 (기본 `Game`은 프로젝트가 변경)

---

## 6. 성능 원칙 (게임 개발 특화)

프레임워크의 예시 코딩 규칙은 모바일 게임 수준의 최적화를 기본으로 합니다:

- GC 할당 최소화 (코루틴, 람다, LINQ, foreach 지양)
- Update에서 무거운 연산 금지
- 풀링 우선
- 런타임 GetComponent 금지

이러한 규칙은 프로젝트의 `CODING_RULES.md`에서 재정의할 수 있습니다.
