# AI 개발 루프 아키텍처 (v1 — SUPERSEDED)

Cursor 워크플로 안에서 동작하는 **AI 개발 루프**의 역할·단계·검증·개선 방식을 정의한다.  
코드 구현 없이 설계만 기술한다.

---

## 1. 루프 개요

한 사이클은 다음 6단계로 구성된다. **안전하고 점진적**으로 진행하기 위해, 계획 → 구현 → 검증 → 요약 순서를 지키고, 실패 시 같은 사이클 안에서 개선 후 재검증한다.

```
  ┌─────────────────────────────────────────────────────────────────┐
  │  1. Read design documents   →   2. Plan feature                 │
  └────────────────────────────┬────────────────────────────────────┘
                               ▼
  ┌─────────────────────────────────────────────────────────────────┐
  │  3. Generate module code    →   4. Run validation                │
  └────────────────────────────┬────────────────────────────────────┘
                               ▼
  ┌─────────────────────────────────────────────────────────────────┐
  │  5. Summarize risk         →   6. Improve if validation fails   │
  └────────────────────────────┬────────────────────────────────────┘
                               │
                     (fail)    │    (pass)
                               ▼
                         다음 기능 / 완료
```

- **입력**: 사용자 요청(기능/버그/개선) + `Docs/ai/` 설계 문서.
- **출력**: 변경된 파일 목록, 리스크 요약, 검증 보고서(통과 시 또는 N회 개선 후 통과 시).

---

## 2. 역할 정의

Cursor에서 **한 에이전트(동일 AI 세션)** 가 모든 역할을 순서대로 수행한다. 역할은 “할 일의 단계”를 나누기 위한 구분이며, 각 단계에서 참조할 문서와 산출물이 다르다.

### 2.1 Planner(플래너)

**역할**: 설계 문서를 읽고, 기능을 모듈에 매핑한 뒤, **구현 계획**을 작성한다.

- **읽는 문서**
  - `Docs/ai/PROJECT_OVERVIEW.md` — 목적, 게임플레이 루프, 기능 도메인, 폴더 구조.
  - `Docs/ai/MODULE_REGISTRY.yaml` — 모듈 경로, 의존성, editable, risk.
  - `Docs/ai/CODING_RULES.md` — 네이밍, MonoBehaviour/런타임 분리, 성능 규칙.
  - (있으면) 해당 기능의 상세 설계 문서(예: `Docs/design/FeatureX.md`).
- **하는 일**
  - 요청된 기능을 **기존 모듈 하나**에 넣을지, **새 모듈**을 만들지 결정.
  - 새 모듈이면 `MODULE_REGISTRY.yaml`에 추가할 항목(name, path, dependencies 등)을 명시.
  - **PLAN** 작성: 단계별 작업 목록, 건드릴 파일/클래스, 사용할 인터페이스·Config·Runtime·Factory·Tests 템플릿.
  - Core/금지 폴더 접근 여부를 명시하고, 접근 필요 시 사용자에게 명시적 허용을 요청.
- **산출물**
  - **PLAN**(마크다운 또는 리스트): 대상 모듈, 신규/수정 파일 목록, 의존 모듈, 위험 요소 한 줄.
- **안전 장치**
  - 계획 없이 코드 작성 금지(autonomous-developer 규칙과 동일). 계획이 승인되거나 “진행해”로 확인된 뒤 Builder 단계로 넘어간다.

### 2.2 Builder(빌더)

**역할**: PLAN에 따라 **모듈 코드만** 생성·수정한다.

- **입력**
  - Planner가 쓴 PLAN.
  - `Docs/ai/CODING_RULES.md`, `MODULE_REGISTRY.yaml`(경로·의존성 준수).
- **하는 일**
  - PLAN에 적힌 순서대로 파일 생성/수정.
  - 새 코드는 **반드시** `Assets/Game/Modules/<ModuleName>/`(및 필요 시 `Shared`) 안에만 둠. Core는 사용자 허용 없이 수정하지 않음.
  - 표준 모듈 템플릿 준수: 인터페이스, Config(ScriptableObject), Runtime(순수 C#), Factory, Tests(최소 1개).
  - 씬/프리팹은 필요한 최소 범위만 변경(대규모 구조 변경 지양).
- **산출물**
  - 변경·추가된 소스 파일 목록.
- **안전 장치**
  - PLAN에 없는 파일/폴더는 만들지 않는다. 범위가 늘어나면 PLAN을 수정한 뒤 다시 Builder 단계를 진행한다.

### 2.3 Reviewer(리뷰어)

**역할**: 검증을 실행하고, 결과를 해석한 뒤 **리스크 요약**을 작성한다. 실패 시 원인을 정리해 “개선” 단계에 넘긴다.

- **입력**
  - Builder가 수정한 파일 목록.
  - **검증 도구 출력**: Unity 메뉴 `Tools/AI/Validate Generated Modules` 실행 결과(검증 보고서 파일).
- **하는 일**
  - 검증 보고서를 읽고, **에러/경고**를 요약(금지 폴더, 구조 위반, 누락된 테스트, 컴파일 에러, 아키텍처 규칙 위반).
  - **리스크 요약** 작성: 변경된 모듈, 영향받는 의존성, risk 수준(low/medium/high), “추가로 확인할 사항” 한두 줄.
  - 검증 **통과**면 루프 종료, 요약만 사용자에게 전달.
  - 검증 **실패**면 원인 목록과 “어떤 규칙을 어겼는지”를 짧게 정리해 **Improve** 단계로 전달.
- **산출물**
  - **Risk summary**: 수정 파일, risk, 주의사항.
  - **Validation summary**: 통과 여부, 에러/경고 개수, (실패 시) 개선 포인트 목록.
- **안전 장치**
  - 검증을 건너뛰지 않는다. 보고서가 없으면 “검증 실행 필요”로 처리하고, 사용자에게 Unity에서 검증 메뉴 실행을 안내한다.

## 3. 루프 단계별 상세

| 단계 | 담당 역할 | 입력 | 산출물 |
|------|-----------|------|--------|
| 1. Read design documents | Planner | 사용자 요청, Docs/ai/*.md, MODULE_REGISTRY.yaml | (내부) 이해된 제약·도메인 |
| 2. Plan feature | Planner | 위 + 기능 요구사항 | PLAN(모듈 매핑, 파일 목록, 위험 요소) |
| 3. Generate module code | Builder | PLAN, CODING_RULES, MODULE_REGISTRY | 변경/추가된 소스 파일 |
| 4. Run validation | Reviewer + (사용자/자동) | 변경된 프로젝트 | 검증 보고서 파일 |
| 5. Summarize risk | Reviewer | 검증 보고서, 변경 파일 목록 | Risk summary, Validation summary |
| 6. Improve if validation fails | Builder | Reviewer가 정리한 실패 원인·규칙 | 수정된 코드 → 4단계로 회귀 |

- **4단계(검증 실행)**  
  - Cursor 안에서는 “Unity 에디터에서 메뉴 실행”이 필요하므로, **사용자가** `Tools/AI/Validate Generated Modules`를 실행하거나, 나중에 CLI/자동화로 실행할 수 있다.  
  - 실행 후 나온 **검증 보고서 파일 경로**를 Reviewer가 참조한다고 가정한다.

- **6단계(개선)**  
  - Reviewer가 적은 “어떤 규칙 위반인지”만 수정한다. PLAN을 벗어나는 새 기능을 이 단계에서 넣지 않는다.  
  - 수정 후 **다시 4단계(검증) → 5단계(요약)** 로 돌아간다. 통과할 때까지 반복하거나, 사용자가 “일단 여기서 중단”할 때까지 반복한다.

## 4. 검증(Validation) 연동

- **위치**: Unity Editor 메뉴 `Tools/AI/Validate Generated Modules` 및 그 결과로 생성되는 **검증 보고서 파일**.
- **검증 항목**(기존 제안과 동일):  
  - 금지 폴더 수정(Core baseline),  
  - 모듈 폴더 구조·필수 파일·Tests 존재,  
  - 컴파일 에러,  
  - 기본 아키텍처 규칙(Runtime/Config 등).
- **루프에서의 사용**
  - **Builder 완료 후**: “검증을 실행해 주세요” 또는 (자동화 시) 스크립트가 검증 실행.
  - **Reviewer**: 보고서 파일을 읽고, 에러가 있으면 “Improve”용 목록(어떤 파일, 어떤 규칙)으로 정리.
  - **개선 후**: 검증을 다시 실행해 통과 여부 확인.
- **안전·점진**: 한 번에 많은 파일을 바꾸지 않고, 한 기능/한 모듈 단위로 검증 → 개선 → 재검증을 반복한다.

## 5. 시간에 따른 개선(Improvement Over Time)

루프가 반복되면서 시스템이 “나아지는” 방식은 다음 네 가지로 한정한다.

1. **문서 정교화**
   - 같은 유형의 실패가 반복되면, `CODING_RULES.md`나 `PROJECT_OVERVIEW.md`에 “자주 하는 실수”나 “AI가 반드시 확인할 것”을 한두 줄씩 추가한다.
   - 새 모듈 패턴이 생기면 `MODULE_REGISTRY.yaml`과 표준 모듈 템플릿 설명을 갱신한다.
   - **주체**: 사용자 또는 AI가 제안하고 사용자가 문서를 수정한다. 문서가 곧 다음 루프의 “설계 문서”가 된다.

2. **검증 규칙 보강**
   - 새로운 위반 패턴이 발견되면, 검증 도구에 **새 검사(Validator)** 를 추가한다(예: 특정 네이밍, 특정 폴더 금지).
   - 기존 루프는 그대로 두고, “4. Run validation”이 더 많은 것을 검사하게 만든다.
   - **주체**: 사용자가 검증 도구 코드/설정을 수정한다.

3. **PLAN 템플릿 정제**
   - Planner가 쓰는 PLAN 형식(필수 항목: 대상 모듈, 파일 목록, Core 접근 여부, 위험 요소)을 `Docs/ai/`에 짧은 템플릿으로 두고, 반복된 실패를 줄이기 위해 “체크리스트”를 추가한다.
   - **주체**: 사용자 또는 AI가 제안하고, 문서를 갱신한다.

4. **역할별 규칙 보강**
   - `.cursor/rules/autonomous-developer.mdc`에 “Planner는 반드시 MODULE_REGISTRY를 먼저 읽을 것”, “Reviewer는 검증 보고서 경로를 명시할 것” 등 한 줄씩 추가할 수 있다.
   - **주체**: 사용자가 Cursor 규칙을 수정한다.

**코드 자체가 스스로 학습하는 구조는 두지 않는다.** 개선은 **문서·검증 규칙·템플릿·Cursor 규칙**을 사람이 수정하는 방식으로만 이루어지며, 그래서 예측 가능하고 안전하게 “다음 루프가 더 잘 지키게” 만든다.

## 6. 안전·점진적 운영 원칙

- **한 사이클 = 한 기능(또는 한 모듈)**  
  - 여러 기능을 한 번에 계획하지 않고, 기능 하나·모듈 하나 단위로 PLAN → Build → Validate → Summarize → (필요 시) Improve를 돌린다.

- **Core/금지 영역**  
  - PLAN 단계에서 “Core 수정 필요 여부”를 반드시 쓴다. 필요하면 사용자 명시적 허용 후에만 Builder가 건드린다.

- **검증 통과 전까지 확정 아님**  
  - “코드만 작성하고 검증 생략”하지 않는다. 검증 실패 시 Improve에서 **해당 위반만** 고치고, PLAN 범위를 임의로 넓히지 않는다.

- **요약은 항상**  
  - 매 사이클 끝에 “변경 파일 + risk + 검증 결과” 요약을 남긴다. 나중에 자동화·CI에서도 같은 요약 형식을 사용할 수 있다.

## 7. Cursor 워크플로에서의 배치

- **단일 세션**: 한 대화 안에서 “플래너 → 빌더 → (검증 실행 안내) → 리뷰어 → 요약” 순서로 진행한다.
- **사용자 액션**: (1) 기능 요청, (2) PLAN 확인·승인, (3) (선택) 검증 메뉴 실행 후 보고서 경로 공유, (4) 실패 시 “개선해줘” 요청 시 6단계만 재실행.
- **규칙과의 관계**: `.cursor/rules/autonomous-developer.mdc`의 “문서 먼저 읽기, PLAN 먼저, 모듈 매핑, Core 금지, 검증 후 요약”과 이 루프가 일치하므로, 같은 규칙을 “단계별로 풀어 쓴 것”으로 보면 된다.

이 문서는 **설계만** 기술한다. 구현(스크립트·에디터 도구·자동화 스크립트)은 별도 단계에서 진행한다.
