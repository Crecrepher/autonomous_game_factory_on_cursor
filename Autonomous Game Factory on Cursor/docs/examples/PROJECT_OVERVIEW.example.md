# 프로젝트 개요

## 1. 프로젝트 목적 및 장르

이 프로젝트는 **아이소메트릭 3D 아케이드 Idle 게임**을 위한 Unity C# 프로젝트이며, **Cursor 기반 자율 AI 보조 개발** 환경을 따른다.

- **아이소메트릭 3D**: 탑다운 시점, 2.5D 또는 3D 오브젝트. 카메라·그리드·깊이 정렬 일관성.
- **아케이드**: 짧은 세션, 명확한 목표, 전투·수집·자동화가 혼합된 액션성.
- **Idle**: 자동 생산, 기계·레일·업그레이드로 점진적 자동화, 재화 누적·소비 루프.

목표는 설계 문서 기반으로 모듈형 게임 시스템을 생성·검증·개선하는 것이다. 실제 게임플레이 루프와 밸런스는 별도 기획 문서에서 정의하며, 이 개요는 **개발 구조·아키텍처·기능 도메인**에 초점을 둔다.

---

## 2. 핵심 게임플레이 루프 (개념)

1. **수집·생산** — 기계에서 아이템 생산, 스택/슬롯 관리, 아이템 레일·자동화.
2. **거래·재화** — 재화 획득/소비, 손님 요구 충족, 가게/경제 루프.
3. **전투** — 조이스틱 이동, 근접/원거리/광역 공격, 투사체·포탑·적 풀링·보스.
4. **성장** — 체력·회복·데미지 상호작용, 무기 슬롯·업그레이드·해금, 전투 레벨.
5. **메타** — 게임 플로우·성공/실패 분기, 가이드·엔드카드·다이나믹 옵션·CTA.

AI 보조 개발 루프: **설계 문서 읽기 → 모듈 생성/수정(MODULE_REGISTRY·CODING_RULES 준수) → 검증 → 개선**.

## 4. 아키텍처 철학

### 4.1 원칙

- **SOLID** — 단일 책임, 개방-폐쇄, 리스코프 치환, 인터페이스 분리, 의존성 역전.
- **의존성 주입** — 생성자/세터/ScriptableObject 등으로 주입, 하드코딩 지양.
- **이벤트 기반 통신** — 필요한 객체끼리만 통신, 전역 이벤트 버스 사용 금지.
- **느슨한 결합·높은 응집** — 모듈 간에는 인터페이스로만 의존, 모듈 내부는 응집도 유지.

### 4.2 구조적 원칙

- **MonoBehaviour는 얇게** — 뷰·바인딩·라이프사이클만, 로직은 서비스·런타임 클래스로 분리.
- **네임스페이스** — 게임 제목(또는 프로젝트 식별자) 기준.
- **ScriptableObject** — 아이템 타입, 생산 규칙, 업그레이드 정보, 보스 패턴 등 설정·데이터.
- **자식만 참조** — 씬/프리팹에서 동일·상위 오브젝트 직접 참조 금지, **자식 오브젝트만** 참조.
